# Каталог иконок и действий Alterios

Этот документ задает рабочий стандарт выбора иконок и поведения действий для форм, списков, групп, процессов и отчетов Alterios.

## Базовый принцип

- Сначала выбирается смысл действия, потом иконка.
- Перед записью в проект вызывается `alterios_ensure_project_icon_library`: tool проверяет registry и файловый менеджер целевого проекта, загружает недостающие SVG из `assets/icons/project-public` и возвращает локальные UUID.
- В `iconId` записывается только UUID файла из текущего проекта. UUID из другого проекта переносить нельзя.
- Видимая кнопка в панели или строке должна быть icon-first. Текст хранится как `title`, tooltip, пункт меню или краткое описание действия.
- Основной порядок действий строки: `edit` -> `preview/view` -> `delete`. Вторичные действия уходят в меню `menu`.

## Правила форм и действий

- Для формы добавления `pageTitle` всегда задается как `Добавить {что добавляем}`.
- Для формы списка `pageTitle` всегда задается как `{наименование сущности во множественном роде}`.
- Для форм просмотра и редактирования `pageTitle` всегда задается как `{наименование сущности в единственном роде}`.
- В форме добавления или редактирования действия страницы идут в порядке: `Закрыть` с `keyboard_return`, затем `Сохранить` с `save`.
- В форме просмотра действие страницы содержит `Закрыть` с `keyboard_return`; сохранение не добавляется.
- `Сохранить` работает по контексту: на конечной форме иерархии это `сохранить -> назад`, на промежуточной форме это только `сохранить`.
- На форме просмотра добавляется переход к редактированию текущей записи через icon-only действие `edit_document` с подсказкой `Редактировать`.
- На форме редактирования переход к редактированию этой же записи не добавляется.
- Действия элемента на формах просмотра и редактирования должны совпадать по составу и смыслу, кроме перехода в редактирование, который есть только на просмотре.
- Для скрипта или обработки используется иконка `forms_apps_script`.
- Если действий элемента больше трех, они группируются через действие `menu`, чтобы не загораживать интерфейс.
- Действия значения в списках по умолчанию группируются через `menu`; обязательный состав: `Редактировать`, `Просмотр`, `Удалить`, все с иконками.
- Для справочника добавляется отдельное действие массового редактирования: форма массового действия показывает поля, разрешенные к выводу, но перед этим проверяется, что выбранные поля не являются связями.
- Если интерфейс уже сделан своеобразно и это похоже на осознанное проектное решение, перед изменением его поведения или визуальной модели нужно уточнить это у пользователя.

## Принцип работы с видами действий

| Вид действия | Смысл | Основная иконка | Где размещать | Обязательная проверка |
|---|---|---|---|---|
| `save` / `submit` | Сохранить текущую форму или запись. | `save` | Нижняя панель формы; после `Закрыть`. | Readback сохраненной записи; на конечной форме иерархии поведение `сохранить -> назад`, на промежуточной форме - сохранить без перехода; если дальше идет script, порядок `submit_all -> manual_script`. |

## Правила размещения

- В главном списке показывать только действия, нужные большинству пользователей: обычно `add_2`, `preview`, `edit`, `menu`.
- Множественный выбор не включать в общий стартовый список. Для bulk-сценария создавать отдельный список или отдельный режим.
- В строке не держать много иконок подряд: основные действия видимы, остальные в `menu`.
- `delete` всегда отделять логически и ставить последним.
- Для форм с сохранением сначала обеспечить понятный заголовок, затем поля, затем действия. Кнопки действия не должны компенсировать непонятную форму.
- Для action со side effect всегда иметь readback: запись, task, process, file metadata, report render или журнал операции.
