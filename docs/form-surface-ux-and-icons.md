# Правила форм, представлений и действий

Дата: 2026-07-10
Область: Alterios/LIMS формы, вкладки, ячейки `view_data_list`/`view_data`,
встроенные отчеты, комментарии и действия строк.

## Цель

Форма должна быть понятной оператору без ручной расшифровки JSON:

- данные из представления занимают логичное рабочее место, без пустых
  визуальных слотов;
- пользователь сканирует экран по F-паттерну: заголовок и ключевой контент
  сверху слева, вторичные блоки ниже или во вкладках;
- роли, источник данных, стили, расположение, жирность, заголовки и названия
  страниц проверяются до записи формы;
- действия строк идут в согласованной логике: `Редактировать`,
  `Просмотр`, `Удалить`;
- действия элемента являются icon-only: видимая кнопка это только иконка,
  текст действия хранится в `tooltip`; `title` внешнего element action должен
  быть пустым. Текст допустим только внутри пунктов вложенного меню.

## Раскладка

1. `pageTitle` обязателен для пользовательской формы. Он должен отвечать на
   вопрос "где я нахожусь", а не повторять технический ID.
   - форма добавления: `Добавить {что добавляем}`;
   - форма списка: `{наименование сущности во множественном роде}`;
   - форма просмотра или редактирования: `{наименование сущности в единственном роде}`.
2. Главный список или карточка на `view_data_list`/`view_data` занимает всю
   строку. Не размещать рядом пустой слот, spacer или случайный блок.
3. Если форме нужны комментарии, отчет или справка, размещать их ниже
   основного контента или отдельной вкладкой с пользовательским названием.
4. Для data-heavy ячеек задавать явную ширину/растяжение, например
   `styles.width = "100%"`, `styles.flex = "1 1 auto"` или аналогичный
   grid/full-width контракт.
5. Не оставлять пустые вкладки и пустые строки. Если блок пока не готов,
   лучше не показывать его в форме.
6. Жирность использовать для заголовков, групп и действительно важных значений.
   Не делать весь список или весь блок жирным.
7. Постоянную текстовую сноску/пояснение под полем использовать только для
   полей, у которых persisted тип контентного поля `date`. Для остальных типов
   пояснение должно уходить в понятный label, tooltip, placeholder или
   отдельный справочный блок, чтобы форма не перегружалась нижним текстом.
   Проверка тестировщика: `alterios_analyze_form_surface` должен не возвращать
   `field_footnote_requires_date`.
8. Если ячейка показывает таблицу (`view_data_list` или аналогичная табличная
   часть), заголовок ячейки должен быть выровнен по центру и иметь жирность
   `bold`/600+.
9. Если отображение не является таблицей, видимый заголовок ячейки не ставится:
   контекст дается через `pageTitle`, название вкладки, label поля или краткую
   подсказку.

## Источники данных

Минимальный контракт ячеек:

| Ячейка | Обязательный источник | Комментарий |
|---|---|---|
| `view_data_list` | `params.viewId` | Для списков, таблиц и связанных представлений. |
| `view_data` | `params.viewId` | Для карточек/деталей по представлению. |
| `content` | `params.contentTypeId` | Для прямой работы с типом контента. |
| `report` | `params.reportId` | Для embedded Stimulsoft-отчета. |
| `comments_list` | `params.openId = true` | Для комментариев текущей записи. |

Для отчета во вкладке текущей записи дополнительно проверять
`params.openId = true` и `POST /api/views/v2/get-data` с top-level
`dataId: ["<openId>"]`. `contentId` сам по себе не считается достаточным
доказательством контекстной фильтрации.

## Роли и доступ

При аудите формы отдельно инвентаризируются ключи, связанные с ролями и
доступом: `roles`, `role`, `permissions`, `access`, `candidateUsers`,
`candidateGroups`. Для BPMN/task-flow форма задачи должна соответствовать
`camunda:formKey`, а доступ к задаче проверяется через assigned/candidate
логику процесса, а не через случайную прямую ссылку на форму.

## Действия и иконки

1. Для иконок использовать Google Fonts Icons.
2. Базовый размер: `16dp`.
3. Базовый цвет: `#4B77D1`.
4. В `cellActionContainers` внешнее действие должно быть icon-only:
   `title = ""`, `tooltip = "{смысл действия}"`, `iconId` ссылается на
   проектную иконку. Нельзя оставлять видимую кнопку `иконка + текст`.
5. В форме добавления или редактирования действия страницы идут в порядке:
   `Закрыть` (`keyboard_return`), затем `Сохранить` (`save`).
6. В форме просмотра действие страницы содержит только `Закрыть` (`keyboard_return`),
   если нет отдельного бизнес-сценария.
7. `Сохранить` на конечной форме иерархии выполняет `сохранить -> назад`, а на
   промежуточной форме только сохраняет без перехода.
8. На форме просмотра добавляется icon-only переход к редактированию текущей
   записи: `edit_document`, tooltip `Редактировать`. На форме редактирования
   этот переход не добавляется.
9. Действия элемента на формах просмотра и редактирования должны совпадать по
   смыслу; различается только наличие перехода к редактированию.
10. Для скрипта или обработки использовать `forms_apps_script`.
11. Если действий элемента больше трех, группировать их через меню. Для меню
    вариантов печати внешний контейнер использует иконку `arrow_drop_down`,
    `title = ""`, `tooltip = "Печать"`, а пункты внутри меню используют иконку
    `print` и человекочитаемые названия вариантов.
12. Действия значения в списках группировать через `menu`; обязательный состав:
    `Редактировать`, `Просмотр`, `Удалить`, все с иконками.
    Проверенный контракт Alterios: внешний `valueActionContainers[]` содержит
    один контейнер `type: "menu"` с иконкой `more_vert`/`menu`, пустым
    `actions: []` и вложенным массивом `containers[]`. Каждый пункт меню является
    отдельным контейнером `type: "action"` с собственными `title`, `iconId` и
    `actions[]`. Не объединять `Редактировать`, `Просмотр`, `Удалить` в один
    контейнер `type: "action"` с несколькими `actions[]`: такой JSON сохраняется,
    но не дает правильного вложенного меню в интерфейсе.
13. Действие по умолчанию для строки списка - `Просмотр`. В JSON это
    `default: true` на вложенном action-контейнере просмотра внутри меню, а не
    на внешнем контейнере `type: "menu"`.
14. Для справочника добавлять отдельное массовое действие только после проверки,
    что поля массового вывода не являются связями.
15. Вторичные действия строки группировать в меню `more_vert`, когда прямых
    действий становится много или они не являются основным сценарием.
16. Порядок row actions:
    - `edit` / `Редактировать`;
    - `visibility` / `Просмотр`;
    - `delete` / `Удалить`.
17. `Удалить` всегда последнее и считается destructive-сценарием. Для него
    нужна отдельная проверка прав, подтверждение или write-gate.
18. Если форма или список уже сделаны в нестандартной, но явно осознанной
    проектной логике, не менять эту модель без уточнения у пользователя.

## Проверка в MCP

Для живой формы:

```python
alterios_analyze_form_surface(
    form_id="<form-id>",
    profile="artx",
    project_id="<project-id>",
)
```

Для локального JSON:

```powershell
alterios-form-surface-check .\form.json --pretty
```

Анализатор возвращает:

- `issues`: конкретные нарушения с `severity`, `code`, `path`;
- `inventory.cell_types`: какие ячейки есть на форме;
- `inventory.data_sources`: привязки `viewId`, `contentTypeId`, `reportId`,
  `openId`;
- `inventory.role_keys`: найденные свойства ролей и доступа;
- `inventory.style_keys`: стили, по которым видно ширину, flex, жирность и
  оформление;
- `inventory.action_icons`: использованные иконки действий.
- `inventory.field_footnotes`: найденные постоянные нижние сноски под полями.

Ошибки `missing_view_source`, `missing_content_type_source` и
`missing_report_source` блокируют качество формы. Предупреждения по пустотам,
иконкам, сноскам и ширине не всегда ломают сохранение JSON, но должны быть исправлены до
боевого использования формы.
