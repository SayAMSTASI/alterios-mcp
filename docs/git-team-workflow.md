# Командная работа и правила push

Этот документ фиксирует границу между разработкой `alterios-mcp`, работой над
бизнесовыми задачами в Alterios и внутренними артефактами Codex.

## Базовый принцип

Git-репозиторий хранит продуктовые изменения MCP: код, тесты, обезличенные
документы, skills, agents и инструкции.

Git-репозиторий не хранит внутренности конкретных бизнесовых задач: реальные
данные проектов, материалы, имена пользователей, сырые HAR, screenshots,
write-journal, inventory snapshots, рабочие заметки, постановки с чувствительным
контекстом и любые секреты.

## Когда Codex может push

Codex может делать commit и push, если задача меняет сам `alterios-mcp`:

- доработка MCP tools, CLI, тестов или проверок;
- исправление бага MCP;
- доработка README, инструкций администратора или пользовательской документации;
- создание, заполнение или переписывание skill/agent по явной просьбе;
- обобщение результата бизнесовой задачи в reusable skill/agent/playbook без
  реальных данных проекта;
- обновление проектных документов команды после завершенного mergeable slice.

Перед push обязательны проверки по типу изменения: targeted tests или полный
`pytest`, `git diff --check`, secret/sensitive scan и ручная проверка списка
измененных файлов.

## Когда Codex не должен push

Codex не делает push, если работа была только бизнесовой или операционной:

- создание или изменение live-объектов в Alterios;
- заполнение материалов, форм, отчетов, задач, пользователей или ролей в
  конкретном проекте;
- исследование конкретного проекта, если результат содержит реальные имена,
  ids, материалы, пользователей, URL или внутреннюю структуру клиента;
- черновики постановок, заметки аналитика, task breakdown, промежуточные
  выводы агентов;
- raw evidence: HAR, screenshots, видео, export-файлы, inventory snapshots,
  write-plans, write-journal.

Если такая работа выявила улучшение MCP, push допускается только для
обезличенной доработки MCP/skill/agent. Внутренний контекст задачи остается
локальным.

## Правило для бизнесовых задач

Если бизнесовая задача требует доработки skills или agents, в git пушится только
универсальная доработка:

- новое правило skill/agent;
- новый reusable сценарий;
- тесты;
- обезличенная документация;
- защитная проверка или validator.

Не пушатся:

- постановка конкретного клиента;
- названия реальных проектов и материалов;
- имена, email, роли, исполнители;
- live ids, ссылки на рабочие пространства, прямые URL контуров;
- рабочие выводы агентов и промежуточные решения.

## Правило для skills и agents

Если пользователь явно просит заполнить, переписать или доработать skill/agent с
учетом правок, это считается изменением продукта. Такой результат коммитится и
пушится после проверок.

Skill/agent должен ссылаться на обезличенные правила и reusable workflow. Если
нужен пример, он должен быть synthetic/example, без данных действующих проектов.

## Ветки и командный процесс

Для командной работы предпочтителен такой поток:

1. Issue или короткая постановка.
2. Ветка `codex/<stage-or-area>-<short-task>`.
3. Изменения только в зоне задачи.
4. Проверки и sensitive scan.
5. Pull request.
6. Review.
7. Merge в `main`.

Прямой push в `main` допустим только в single-owner режиме или для маленьких
документированных правок после проверок.

## Что хранить локально

Локально, вне git, хранятся:

- `.env` и любые `.env.*`;
- `artifacts/`;
- raw HAR и screenshots;
- inventories и write journals;
- временные постановки;
- протоколы работы агентов;
- local private workboard: `index.md`, `issues/*`, `sprints/*`, `evidence/*`;
- выгрузки из Alterios;
- evidence с реальными данными.

Если нужно передать evidence команде, его сначала обезличивают или переносят в
закрытое хранилище с отдельными правилами доступа.

## Итоговый отчет Codex после push

После push Codex должен сообщить:

- commit hash;
- какие файлы изменены;
- какие проверки прошли;
- выполнялась ли live-запись;
- были ли обнаружены чувствительные данные;
- что осталось локальным и не было отправлено в git.
