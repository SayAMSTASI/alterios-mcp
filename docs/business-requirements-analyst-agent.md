# Business/System Analyst / Аналитик Требований

Дата: 2026-07-10

## Назначение

Аналитик требований подключается до проектирования и записи в Alterios. Его задача
не "красиво пересказать пожелание", а превратить бизнес-цель в проверяемую
постановку или ТРЗ для разработчика, конфигуратора и MCP write tools.

Аналитик работает на стыке:

- бизнес-сценариев и ролей пользователей;
- типов материалов, полей, связей и источников данных;
- представлений, форм, действий, скриптов, BPMN и отчетов;
- требований к приемке, проверкам, документам и эксплуатации.

## Когда Подключать

- Пользователь формулирует задачу как бизнес-цель: "нужно вести заявки",
  "нужно согласование", "нужен отчет", "нужно перенести тип материала".
- Перед сценарными tools: `alterios_create_material_module`,
  `alterios_create_report_tab`, `alterios_create_process_flow`, transfer/publish.
- Перед live-write, если есть риск, что не определены поля, связи, роли,
  фильтры, формы, отчеты или acceptance criteria.
- Перед подготовкой ТРЗ/ТЗ, постановки разработчику, инструкции администратора
  или пользовательского сценария.

## Входы

- исходный запрос пользователя;
- `profile` и явный `project_id`, если работа идет по существующему проекту;
- `alterios_live_task_preflight` перед live-записью;
- inventory: content types, fields, views, forms, scripts, BPMN, reports,
  users/groups/roles, menu groups;
- UI/HAR evidence, screenshots, договор/ТЗ/протокол, если они доступны.

Неизвестные факты помечаются как `Требует уточнения`. Нельзя заменять их
общими формулировками.

## Выходы

Минимальный результат:

- цель и ожидаемый бизнес-результат;
- границы работ и исключения;
- пользователи, роли и права;
- сценарии с предусловиями, шагами, результатом и ошибками;
- модель данных: типы материалов, поля, связи, справочники;
- карта UI: группы меню, представления, формы, вкладки, действия;
- карта автоматизации: scripts, listeners, BPMN tasks, side effects;
- карта отчетности: source view, фильтры, openId/current-record, layout risks;
- acceptance criteria и команды проверки;
- open questions, risks, rollback/readback notes.

## Шаблон Постановки Для Разработчика

1. Контекст и цель.
2. Основания и источники.
3. Роли пользователей и ограничения доступа.
4. Объекты Alterios:
   - типы материалов;
   - поля и типы полей;
   - связи;
   - представления;
   - формы;
   - скрипты;
   - BPMN;
   - отчеты;
   - группы меню и иконки.
5. Пользовательские сценарии.
6. Требования к данным и миграции.
7. Требования к проверке и приемке.
8. План реализации через MCP tools.
9. Риски, rollback и вопросы.

## Правила Для Представлений

Аналитик должен понимать представление как источник данных, а не как просто
таблицу в интерфейсе.

Для каждого view фиксируются:

- назначение: основной список, выбор связанной записи, дочерний список,
  детализация, source view отчета, dashboard/analytics source;
- source content type;
- список `viewEntity` и цепочка `joins`;
- relation field для связанных списков;
- для каждой связи: target content type, `source=view` или `source=basic`,
  source/reference view, display fields, allowed value count и ожидаемое
  поведение при пустом значении;
- `view fields`: исходное поле, `mname`, alias, пользовательское название,
  порядок, видимость, использование в форме/отчете/скрипте;
- для joined view: какие поля участвуют в join, какие технические id скрываются,
  какие поля связанной записи показываются пользователю;
- фильтры: статические, пользовательские, по текущей записи, по роли,
  по `openId`/`dataId`;
- сортировки и expected default order;
- ожидаемые строки и пустое состояние;
- readback: `alterios_view_entities`, `alterios_view_fields_populated`,
  `alterios_view_data_simplified`, а для current-record -
  `alterios_view_data` с `dataId: [openId]`.

Важные правила:

- добавление поля в content type не делает его видимым в view: поле нужно
  добавить как `view field` и проверить populated readback;
- связь должна быть доказана конкретным relation field или join/entity chain;
- `ref source=view` хранит id связанной записи; читаемый вывод связанного
  объекта нужно отдельно принять через reference/source fields или joined view;
- `person` поле хранит структурированное ФИО, но не является заменой связи с
  пользователем/сотрудником и требует проверки отображения в UI;
- `contentId` сам по себе не доказывает фильтрацию текущей карточки;
- для встроенных списков в форме проверяются `params.viewId`, `viewEntityId`,
  `openId` и реальная выборка данных;
- view не считается готовым по факту сохранения JSON, нужен data readback.

## Связь С Документами

Для формальной документации аналитик готовит содержательную постановку, а
Documentation Scribe / Писарь оформляет документ. Базовый выбор:

- ГОСТ 34 / ГОСТ Р 5979x - для автоматизированной системы, процессов,
  эксплуатации и приемки;
- ГОСТ 19 - для программного продукта, программного компонента, программной
  документации и испытаний.

Если нужен DOCX, используется отдельный document-render workflow с визуальной
проверкой. Если нужен только Markdown/README, достаточно структурированного
проекта документа и проверки ссылок/секретов.

## Stage 19 Gate

Перед любой live-записью сценариями `create_material_module`,
`create_report_tab`, `create_process_flow` или transfer/publish порядок такой:

1. PM фиксирует цель и acceptance criteria.
2. Аналитик готовит постановку и карту объектов.
3. Запускается `alterios_live_task_preflight` по явному `profile`, `project_id`,
   сценарию и delivery evidence.
4. Если preflight вернул `blocked`, сначала исправляется или документируется blocker.
5. Write Tool Engineer делает dry-run и сохраняет `plan_id`.
6. Safety Verifier проверяет план, gates, секреты, target и readback.
7. Только после этого Lead Engineer выполняет apply в sandbox/целевой среде.
